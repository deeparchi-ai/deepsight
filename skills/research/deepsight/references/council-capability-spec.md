# Council 能力声明（厂商无关）

> 把"审计需要什么样的模型"和"今天恰好能用哪个型号"分开。点名型号（DeepSeek V4 Pro / Kimi K2.6 / Gemma 4 …）高度不稳定——`council-model-setup.md` 自己的实测记录就显示"同一天上午有、下午没"。**不要把型号写死进方法论。** 本文件定义能力要求；具体型号见 `council-model-setup.md`，且只当作**可随时替换的快照**。

## 审计席位的能力要求（不是型号）

| 维度 | 要求 |
|------|------|
| 模型族多样性 | ≥3 个独立 model family（定义见 `council-protocol.md` §1）；同族不同尺寸不算 |
| 上下文 | 每个审计模型 ≥ 待审材料 token 数（建议 ≥128K；长材料 ≥256K） |
| 可达性 | Council 执行前 curl 实测连通，**不依赖文档记录的"可用"状态** |
| 成本 | 审计可用免费层/低成本模型；主推理与审计预算分离 |
| 角色匹配 | 5 角色尽量分配到不同模型族，避免同族占多个角色 |

## 选型流程（运行时决定，不写死）

1. 列出当前**可达**模型（curl 探测），按 model family 分组。
2. 独立族数 `D≥3` → 正常执行；否则按 `council-protocol.md` §2 **诚实降级**（封顶证据级 + 报告标注），不得静默跳过。
3. 角色分配：优先让"唱反调者 / 第一性原理"用与主模型**差异最大**的族。
4. 把实际所选 `{角色 → 模型族 → 型号}` 写入 `council_provenance`。

## 为什么不写死型号

- 免费层可用性以小时计变化、限流频繁（实测见 `council-model-setup.md`）。
- 写死型号 = 方法论绑定某一时刻的供给现状，迁移即失效。
- 能力声明可移植：任何满足"≥3 独立族 + 足够上下文 + 可达"的组合都能跑 Council。

具体可用型号 / 接入配置（易过期的快照）：见 [`council-model-setup.md`](council-model-setup.md)。
