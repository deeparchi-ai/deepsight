# 轻量模式走查（示例）

> **这是一份示范，不是权威结论。** 故意挑了一个小而可外部查证的技术问题，让 `[A/B/C]` 证据分级能诚实落地。目的是展示 DeepSight 轻量模式的**机器如何运转**，不是给出投资级判断。

**问题：** 单机 Agent 的本地状态存储，选 **SQLite** 还是嵌入式 KV（**LMDB**）？

---

## Phase 0 · 路由

- 模式：**轻量**（单一维度的技术选型对比，非真金白银决策）。
- 框架：**技术评估**（信号词："技术路线""选型"）。四步压缩为轻量版：核心需求 → 候选对比 → 风险 → 建议。

## Phase 1 · 约束追问（取技术链前 2 轮）

1. **物理/机制约束：** 访问模式是什么？——Agent 状态多为"少量结构化记录 + 偶发区间查询/二级索引"（会话、turn、tool 调用）。→ 需要**查询能力**，不只是 key→value 取值。
2. **工程约束：** 并发与崩溃语义？——单机但可能多进程/多线程（gateway + cron + CLI）写同库；崩溃后状态必须一致可恢复。→ 需要**事务 + 明确并发模型**。

## Phase 1 · 对抗审查（工程/技术模板，节选）

- *最弱假设：* "KV 更快所以更好"——把性能当唯一轴，忽略查询/事务需求。
- *反面证据：* LMDB 是 `key→value`，二级索引/区间查询要**应用层自己实现**；多进程写需自管锁策略。
- *置信度校准：* 对"SQLite 更契合"信心 8/10；对"具体 QPS 数字"信心 3/10（依赖硬件/schema，不敢量化）。

## Phase 1 · 证据分级结论

- `[A-验证]` SQLite 提供 SQL 查询、二级索引、ACID 事务，单写多读、WAL 模式支持并发读——官方文档可查证。
- `[A-验证]` LMDB 是内存映射 B+tree KV，读极快、单写者事务，但**无 SQL/二级索引**，区间/二级检索需应用层实现。
- `[B-推理]` Agent 状态以"结构化 + 偶发关系查询"为主 → SQLite 的查询/事务模型与需求更契合，省掉自建索引层。
- `[C-推断]` 在典型 Agent 写入量级（远低于 LMDB/SQLite 的性能上限）下，二者性能差异**不构成选型决定因素**——属经验类推，未做基准。

> `[C]` 占比 1/4 ≈ 25%，在 20–35% 目标区间内。✅

## 决策刹车 + 建议

- **建议（可行动，基于 [A]+[B]）：** 选 **SQLite**——查询/事务/索引开箱即用，契合 Agent 状态的结构化+关系特征。
- **`[C]` 不进硬建议：** "性能差异可忽略"降级为**待验证**——
  > **翻盘条件：** 若实际为超高频 KV 读写（如把它当热点缓存、写入达万级 QPS），则 LMDB 的内存映射读优势可能反转结论——届时需基准测试再定。
